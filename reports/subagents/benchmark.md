## Key Findings

- No runtime MCP server implementation exists in the repository yet, so dogfood
  execution can only be scaffolded and should report `INCONCLUSIVE` until a live
  local HTTP MCP server is available.
- The deterministic dogfood path must avoid native Codex file/shell/git bypasses
  during task execution. The runner therefore prepares fixtures and optionally
  starts a server, then performs the coding loop exclusively through
  `initialize`, `tools/list`, and `tools/call`.
- SWE-bench requires Docker, the `swebench` package, substantial disk, and real
  generated predictions. Placeholder predictions are useful for schema and
  command scaffolding but cannot support a pass claim.
- The smoke subset currently validates against the SWE-bench Lite dataset rows
  API. The local environment does not have Docker or the `swebench` Python
  package, so official harness evaluation was not run.

## Concrete Recommendations

- Wire the implementation server to expose the required P0 tool names and JSON
  schemas, then run `benchmarks/dogfood/mcp_deterministic_runner.py` with a real
  `--server-command`.
- Replace placeholder SWE-bench predictions with patches generated by the native
  baseline runner and the MCP-only candidate runner before invoking official
  evaluation.
- Keep benchmark reports honest: use `PASS` only after resolved counts are
  parsed from official results and
  `candidate_mcp_resolved >= baseline_native_resolved`.

## Risks

- Tool argument schemas may differ from the profile. The dogfood runner adapts
  common field names, but incompatible schemas should be treated as benchmark
  failures rather than silently bypassed.
- Empty-patch SWE-bench predictions are intentionally not a valid regression
  comparison.
- A local HTTP server that logs JSON-RPC noise into responses will break the
  runner, which is desirable because stdout pollution is a protocol bug.
